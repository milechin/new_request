# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working in this request workspace.

## What this is

This is an **HPC troubleshooting request workspace** on the Boston University SCC, created by `new_request.sh`. An HPC facilitator uses it to reproduce and debug a researcher's R issue in isolation from their home directory. Your job is to help diagnose and fix the researcher's problem using the context gathered here.

## Start here

1. Read everything in **`context/`** — the facilitator's problem description (`context/problem.md`) and relevant links (`context/links.md`).
2. If **`context/SUMMARY.md`** exists, read it first — it is the overview produced by the `/init-request` command (problem, environment, scripts, links, open questions).
3. If it does not exist, run **`/init-request`** to build it.

## Directory structure

- **`scripts/`** — the researcher's scripts (copied in by the facilitator). This is the code under investigation.
- **`data/`** — input data used by the scripts. *(gitignored — may be large)*
- **`output/`** — output the scripts produce. *(gitignored)*
- **`context/`** — facilitator-authored context: `problem.md` (the issue), `links.md` (tickets/docs), `SUMMARY.md` (generated by `/init-request`), and **`logs/`** — researcher-uploaded build/job logs (*gitignored; the dir is kept via `.gitkeep`*). `/init-request` samples each log's head + tail (and triages build logs) rather than reading them whole.
- **`env_setup/`** — holds `renv.lock` (manifest of the reproduced R package versions, *tracked*) and `.renv-tools/` (the renv install used to generate it, *gitignored*). The R helper scripts themselves are **not** here — they are central scripts in the new_request repo's `bin/r/` (`r_env.sh` setup, `r_snapshot.sh` manifest, `r_install.sh` reproduce-an-install), run against this workspace.
- **`module_load.sh`** — **the activation script**; `source module_load.sh` to load the R module, set `R_LIBS_USER`, put the request's `bin/<lang>` tool(s) on `PATH`, and redirect caches/config into the workspace. *(gitignored)*
- **`R/<version>/`** — the reproduced R package library for that R module. *(gitignored — large)*

## Tools & assets available here

From the symlinked `.claude/` and (once you `source module_load.sh`) the `bin/<lang>` toolset on PATH:
- **`/init-request`** — build/refresh `context/SUMMARY.md`.
- **`triage-build-log` skill** — analyze a build/compile/install log and classify the failure (supports R, Python, C/C++, Fortran). Backed by `triage_build_log.sh`.
- **`r-install-debugger` agent** — reproduce a failing R package install and return a classified verdict (uses `r_install.sh`).

## Working conventions

- **Activate the environment before running anything R-related:** `source module_load.sh`. This also points `XDG_*`, `RENV_PATHS_ROOT`, and `R_ENVIRON_USER`/`R_PROFILE_USER`/`R_HISTFILE` into this workspace, so package caches/config/data stay here instead of the facilitator's home directory.
- The reproduced library is a **faithful copy** of the researcher's environment (same cluster, same R version/arch) — treat package versions as fixed evidence, not something to "fix" by upgrading, unless that is the diagnosis.
- `renv.lock` is the audit record of what was reproduced; consult it to reason about package versions.
- Don't write outside this workspace; don't modify the facilitator's home directory.
